# 產品需求文件 (PRD)：線上宮廟抽籤與捐獻系統

## 1. 產品概述 (Product Overview)
*   **專案背景 (Background)**：
    現代人生活節奏快，需要心靈上的慰藉與指引，但往往無暇親自前往實體廟宇。為了延續民間信仰的傳統，本專案旨在提供線上的「宮廟抽籤系統」，並進一步整合了「抽籤紀錄」與「線上捐獻香火錢」的功能，打造一個完整的線上參拜與回饋體驗。
*   **產品目標與價值 (Objectives & Value)**：
    *   提供具備沉浸感與儀式感的線上抽籤與擲筊體驗。
    *   幫助使用者追蹤、回顧過去求得的籤詩，觀察神明在人生不同階段給予的指引。
    *   提供便捷的線上添油香（捐獻）管道，建立與使用者的雙向互信與回饋循環。
*   **目標受眾 (Target Audience)**：
    *   有傳統信仰，喜歡透過求神問卜尋求人生指引的群眾。
    *   面臨抉擇或壓力，需要心靈寄託的年輕世代與上班族。

## 2. 功能需求 (Functional Requirements)
*   **功能清單與優先級 (Feature List & Priorities)**：
    *   **P0 (核心功能)**：
        *   **抽籤與解析機制**：線上求籤、擲筊（可選），並顯示白話文解說。
        *   **使用者識別**：簡易的帳號或名稱登入綁定機制，作為資料庫儲存的依據。
        *   **抽籤紀錄儲存與查詢**：使用者可進入個人頁面查看過去抽過的籤詩紀錄、求問日期與問題。
    *   **P1 (進階功能)**：
        *   **線上捐獻香火錢**：提供金額選項（例如 100, 500, 1000 元）。初期開發可以「模擬捐款」處理，單純將紀錄存入系統中。
        *   **捐獻芳名錄 / 感謝狀**：完成捐款後給予線上感謝狀或紀錄。
*   **使用者故事 (User Stories)**：
    *   身為一個求籤者，我想要在抽出籤詩後，系統能**自動將抽籤紀錄與我的提問儲存下來**，以便於我日後可以隨時回顧神明的指示。
    *   身為一個還願者或想表達敬意的使用者，我想要**透過網站捐獻香火錢**，以便於在線上也能即時答謝神明。
    *   身為一個使用者，我想要能查看我過去完整的「捐獻紀錄」與「歷史籤詩」，以便於管理我的線上參拜歷程。
*   **業務規則與邊界條件 (Business Rules & Edge Cases)**：
    *   **資料儲存**：所有的抽籤歷程與捐款紀錄，都必須基於關聯式資料庫妥善紀錄對應用戶 ID。
    *   **歷史紀錄權限**：必須透過正確的識別（User ID 或 Session）才能調閱個人專屬的歷史抽籤與捐獻紀錄，確保隱私。

## 3. 使用者體驗與流程 (UX/UI & Flow)
*   **使用者流程 (User Flow)**：
    1.  **首頁/識別登入**：使用者輸入名稱或登入，進入虛擬宮廟大廳。
    2.  **主功能選擇**：畫面提供三個主要入口：「求神問卜」、「香火捐獻」、「我的參拜紀錄」。
    3.  **抽籤流程**：輸入心中想問的事 -> 系統產生籤號 -> 提供籤詩解析 -> 自動寫入資料庫。
    4.  **捐獻流程**：進入捐獻頁 -> 選擇欲供奉金額與祈福項目 -> 確認送出表單 -> 顯示感謝狀並記錄至資料庫。
    5.  **查看紀錄**：進入紀錄頁面，列表中呈現時間、行為（抽籤/捐款）及詳細結果。
*   **介面需求 (UI Requirements)**：
    *   **風格**：現代東方文化風格，兼具莊嚴靜謐感與現代 UI 的簡潔。
    *   **技術介面**：依據此專案技術指定，前端預計採用純靜態的 HTML 結合 Vanilla CSS/JS 打造介面。

## 4. 非功能性需求 (Non-Functional Requirements)
*   **效能與架構考量 (Performance & Architecture)**：
    *   前端採取 HTML，後端則使用 FastAPI 架設 API，並搭配 SQLite 做為輕量化且符合關聯式的本地資料庫。
    *   確保前端在發送表單 (Fetch/Axios) 寫入抽籤或捐款紀錄時的反應時間順暢。
*   **安全性 (Security)**：
    *   由於引入了資料庫 (SQLite)，必須透過 FastAPI 嚴格防範 SQL Injection，確保使用者寫入的神明提問與表單資料安全。

## 5. 數據追蹤與驗收標準 (Metrics & Acceptance)
*   **驗收標準 (Acceptance Criteria)**：
    *   使用者順利完成一次抽籤後，可以在 SQLite 資料庫中正確查到該筆 Log。
    *   使用者可以在畫面的「歷史紀錄」清單中，看到由後端 API 撈取的歷史紀錄。
    *   使用者送出香火錢捐獻表單後，能在資料庫產生對應的捐款明細。
*   **排除範圍 (Out of Scope)**：
    *   真實的綠界/藍新信用卡金流串接（本次專案僅於 SQLite 中做「捐款單據」模擬寫入與建立流程）。
    *   複雜的角色權限區分 (RBAC) 或宮廟後台管理儀表板。
